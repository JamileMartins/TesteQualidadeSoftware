CPRE - Capítulos 5, 6 & 7

Do Caos ao Controle

Como Processos, Gestão e Ferramentas evitam que os projetos de software implodam.

Já temos os requisitos. E agora?

Nas aulas anteriores, aprendemos a elicitar, analisar e documentar requisitos.

Mas os requisitos não são estáticos. Eles mudam. Os stakeholders discordam. O mercado evolui.

Sem uma estrutura para gerir esta complexidade, o projeto entra em colapso. É aqui que entram os Processos e a Gestão de Requisitos.

Capítulo 5

Processo de ER: Não existe "processo único"

Um processo de Engenharia de Requisitos (ER) deve ser adaptado ao projeto. A escolha depende de fatores cruciais:

Stakeholders Estão sempre disponíveis ou só no início do projeto?
Complexidade É um sistema crítico (ex: aviação) ou apenas um MVP para validar uma ideia?
Volatilidade Os requisitos são estáveis ou espera-se que mudem constantemente?

As 3 Dimensões para Configurar o seu Processo

Tempo
Sequencial (Cascata)

Requisitos definidos no início. Pouca flexibilidade.

Iterativo (Ágil)

Requisitos evoluem em ciclos. Alta flexibilidade.

Finalidade
Prescritivo

A especificação é um contrato rígido. Tudo deve ser implementado.

Exploratório

Os objetivos são conhecidos, mas os requisitos são descobertos.

Alvo
Cliente Específico

Software sob medida para um único cliente.

Orientado ao Mercado

Produto para um segmento de mercado (ex: App na Play Store).

Exemplos de Processos no Mundo Real

Processo Contratual

Combinação: Sequencial + Prescritivo + Cliente Específico.

Cenário Típico: Projetos governamentais, outsourcing com escopo fixo. A documentação (SRS) é a base do contrato.

Processo Participativo (Ágil)

Combinação: Iterativo + Exploratório + Cliente Específico.

Cenário Típico: Uma equipe Scrum desenvolve um sistema interno, com o Product Owner (cliente) colaborando diariamente.

Processo de Produto

Combinação: Iterativo + Exploratório + Orientado ao Mercado.

Cenário Típico: Uma startup a construir um novo SaaS (Software as a Service). Os requisitos evoluem com o feedback dos primeiros utilizadores (beta testers).

Capítulo 6

Gestão de Requisitos: A Arte de Manter a Sanidade

Gerir requisitos é o processo contínuo de armazenar, alterar e rastrear os requisitos ao longo do tempo. É o sistema imune do projeto contra o caos.

Prática 1: Gestão do Ciclo de Vida

Um requisito não é apenas "feito". Ele tem um ciclo de vida, e cada estado importa.

Exemplo de um Fluxo de Estado Simples:

Rascunho
➡️
Em Revisão
➡️
Aprovado
➡️
Implementado
➡️
Verificado

No dia a dia: Em ferramentas como Jira ou Azure DevOps, isto é o "workflow" de uma User Story. Saber o estado de cada requisito evita que a equipe trabalhe em itens não aprovados ou teste funcionalidades que ainda estão em rascunho.

Prática 2: Controle de Versão e Baselines

Controle de Versão

Assim como o código (`git`), os requisitos precisam de versões. Cada mudança gera uma nova versão, permitindo rastrear o histórico e reverter se necessário.

No dia a dia: "Porque é que esta regra de cálculo mudou? Quem a aprovou?" O histórico de versões da página no Confluence ou do item no Jira responde a isso.

Baseline

É uma "fotografia" estável e aprovada de um conjunto de requisitos num determinado momento (ex: "baseline da Versão 1.0").

No dia a dia: Antes de iniciar uma Sprint ou uma nova fase de desenvolvimento, cria-se uma baseline. Qualquer mudança a partir desse ponto precisa de seguir um processo formal de gestão de mudança, protegendo a equipe do "scope creep" (aumento descontrolado do escopo).

Prática 3: Rastreabilidade - O Fio de Ouro da Qualidade

Rastreabilidade é a capacidade de seguir a vida de um requisito, para a frente e para trás, conectando-o a tudo o que lhe diz respeito.

Para Trás (Origem)

  • Qual objetivo de negócio justifica este requisito?
  • Qual stakeholder o solicitou?
  • De qual documento fonte ele veio?

Para a Frente (Implementação)

  • Qual User Story ou item de backlog o detalha?
  • Qual parte do código o implementa?
  • Qual caso de teste o valida?

Rastreabilidade na Prática

Exemplo de uma Cadeia de Rastreabilidade:

Objetivo de Negócio

"Aumentar vendas em 10%"

🔗
Requisito

"Permitir pagamento com MB WAY"

🔗
User Story #123
🔗
Código (Commit XYZ)
🔗
Caso de Teste #456

No dia a dia: Quando um teste falha, a rastreabilidade permite identificar qual requisito e objetivo de negócio foram impactados. Quando o negócio questiona a necessidade de uma funcionalidade, a rastreabilidade mostra qual objetivo ela suporta.

Prática 4: Lidar com a Mudança

Mudar um requisito não é só "editar o texto". Exige um processo para avaliar o impacto.

Abordagem Formal (Cascata)

A mudança é proposta através de um "Pedido de Mudança" (Change Request). Um comité (Change Control Board - CCB) reúne-se para analisar o impacto no custo, prazo e risco antes de aprovar ou rejeitar.

Abordagem Ágil

A mudança é adicionada como um novo item no Product Backlog. O Product Owner é o responsável por a priorizar contra os outros itens. A mudança não interrompe a Sprint atual; entra na fila para uma Sprint futura.

Prática 5: Priorização

Nenhuma equipe tem recursos infinitos. Priorizar é decidir "o que fazer agora".

Técnica Simples: MoSCoW

M - Must-Have: Essencial. Sem isto, a entrega não tem valor.
S - Should-Have: Importante, mas não vital. A solução funciona sem isto.
C - Could-Have: Desejável, mas de baixo impacto. "Nice to have".
W - Won't-Have (this time): Não será feito nesta entrega, mas pode ser considerado no futuro.

No dia a dia: Durante o planeamento de uma Sprint ou de uma Release, classificar os itens com MoSCoW ajuda a focar a equipe no que realmente agrega valor e a gerir as expectativas dos stakeholders.

Prática 6: Atributos e Visualizações

Metadados que enriquecem os requisitos e permitem filtrá-los de forma inteligente.

Atributos

São "etiquetas" ou campos associados a um requisito: Prioridade, Risco, Sprint, Módulo Afetado, Autor, Fonte (Stakeholder), etc.

Visualizações (Views)

São simplesmente filtros aplicados sobre os atributos para criar listas úteis.

No dia a dia:

  • "Mostre-me todos os requisitos com Prioridade 'Alta' para o Módulo 'Pagamentos'."
  • "Crie um relatório com todos os requisitos solicitados pelo Stakeholder 'Marketing'."

    Ferramentas: Automatizar o Controle

    Usar as práticas anteriores em ficheiros Word ou Excel é possível, mas insustentável em projetos reais. As ferramentas existem para suportar o processo.

    Tipos de Ferramentas de ER

    Gestão de Requisitos

    Focadas em armazenar, rastrear e gerir o ciclo de vida. São a "base de dados" dos requisitos.

    Modelagem

    Ajudam a criar diagramas (UML, BPMN) para visualizar processos e estruturas, complementando a documentação textual.

    Colaboração e Documentação

    Plataformas para criar e partilhar conhecimento, como wikis, onde a documentação mais detalhada pode residir.

    Exemplos de Ferramentas no Mercado (1/2)

    Para Ambientes Ágeis (Foco em Backlog)

    Jira: O padrão da indústria para gestão de tarefas e backlogs. Excelente para rastreabilidade de User Stories.

    Azure DevOps: A solução integrada da Microsoft. Combina gestão, repositórios de código e pipelines de CI/CD.

    Para Documentação e Colaboração

    Confluence: Uma wiki corporativa que se integra perfeitamente com o Jira, ideal para a documentação detalhada que não cabe numa User Story.

    Exemplos de Ferramentas no Mercado (2/2)

    Para Gestão Formal (Foco em Documento)

    Jama Connect / IBM DOORS Next: Ferramentas robustas para indústrias reguladas (automobilística, médica) que exigem rastreabilidade rigorosa e gestão de baselines complexas.

    Para Modelação Visual

    Lucidchart / draw.io: Ferramentas para criar diagramas de fluxo, UML, BPMN, etc., que ajudam a visualizar e a validar os requisitos de forma colaborativa.

    A Regra de Ouro das Ferramentas

    O Processo vem ANTES da Ferramenta.

    Uma ferramenta não vai resolver um processo caótico. Ela vai apenas automatizar o caos.

    1. Defina o seu processo: Como vai gerir mudanças? Como vai priorizar?
    2. Identifique os problemas: Onde é que o seu processo manual está falhando?
    3. Escolha uma ferramenta que resolva esses problemas.
    4. Comece com um projeto piloto antes de a impor a toda a organização.

    Resumo da Ópera

    Processo (Cap. 5): É o mapa. Adapte-o ao terreno do seu projeto (ágil, cascata, etc.).

    Gestão (Cap. 6): São as regras de trânsito. Rastreabilidade, controle de versão e gestão de mudanças mantêm o projeto a andar de forma segura.

    Ferramentas (Cap. 7): É o veículo. Escolha um que se adeque ao seu mapa e às suas regras, não o contrário.

    A sua Missão como Engenheiro de Software

    Uma boa gestão de requisitos não é burocracia.

    É o mecanismo que impede os projetos de falharem sob o peso da complexidade e da mudança inevitável.

    O seu trabalho não é apenas construir o software, mas garantir que o software certo é construído, e que ele pode evoluir de forma controlada.
