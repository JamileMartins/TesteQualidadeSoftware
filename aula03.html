CPRE CAP. 3 - PRODUTOS DE TRABALHO

O Que São os "Artefatos" da Engenharia de Requisitos?

Um "Produto de Trabalho" é qualquer resultado gerado no processo, desde um rabisco num guardanapo até um documento formal de 500 páginas. Eles são a forma como capturamos, comunicamos e validamos o entendimento.

Fundamento #1: Linguagem Natural

Vantagens

É expressiva, flexível e todos a entendem sem treino. É o ponto de partida.

Desvantagens

É terrivelmente ambígua. "O sistema deve ser rápido." Rápido quanto? Para quem? Em que condições?

Fundamento #2: Templates (A Solução para a Ambiguidade)

Templates fornecem uma estrutura pré-definida para superar as deficiências da linguagem natural. Eles nos forçam a pensar nos detalhes.

Exemplo Clássico (User Story):

Como um [Tipo de Usuário], eu quero [Realizar uma Ação] para que eu possa [Obter um Benefício].

Isso nos força a definir o QUEM, o QUÊ e o PORQUÊ.

Fundamento #3: Modelos (Uma Imagem Vale Mais que Mil Requisitos)

Modelos são representações abstratas para entender relações complexas. Eles nos ajudam a ter uma visão geral e a descobrir inconsistências que o texto esconde.

  • Diagramas de Contexto: Mostram o sistema e suas fronteiras.
  • Diagramas de Classe (UML): Modelam a estrutura e os dados.
  • Diagramas de Atividade (UML): Modelam fluxos e processos.

Exemplo Prático: Modelagem de Contexto

👤

Cliente

↔️

Sistema de E-commerce

↔️
💳

Gateway de Pagamento

👤

Cliente

⬇️

Sistema de E-commerce

⬇️
💳

Gateway de Pagamento

Este modelo simples define o escopo. O que está dentro do sistema? Com quem ele conversa? Isso evita o "scope creep" (aumento descontrolado do escopo).

O Dicionário do Projeto: O Glossário

"Cliente" para o marketing pode ser diferente de "Cliente" para o financeiro. Um glossário centralizado garante que todos falem a mesma língua, evitando mal-entendidos caríssimos.

Se não há um glossário, o projeto está fadado a falhas de comunicação.

O BLUEPRINT DO PROJETO

A Estrutura de Documentação (SRS)

Juntar tudo isso de forma aleatória não funciona. Precisamos de uma estrutura. A mais famosa é a Especificação de Requisitos de Software (SRS), padronizada pela IEEE 830.

O Papel do Padrão IEEE 830

O IEEE 830 não é só um template, é um guia para garantir a qualidade do documento de requisitos. Um bom SRS, segundo o padrão, deve ser:

Correto
Completo
Consistente
Inequívoco
Verificável
Modificável
Rastreável

Tangibilizando Ideias: Protótipos

Às vezes, a melhor forma de validar um requisito é mostrando-o. Protótipos (de baixa ou alta fidelidade) são uma ferramenta poderosa para obter feedback rápido e evitar a construção de algo que o usuário não quer.

É mais barato mudar um protótipo do que mudar o código em produção.

CPRE CAP. 4 - ELABORAÇÃO DE REQUISITOS

O Processo de "Escavar" a Informação Correta

Documentar é importante, mas de nada adianta se a informação estiver errada ou incompleta. A elaboração é o processo ativo de encontrar, negociar e validar os requisitos.

Passo 1: Fontes - Onde os Requisitos se Escondem?

Eles não aparecem magicamente. Precisamos buscá-los ativamente em:

👥

Stakeholders

Usuários, gestores, clientes, desenvolvedores...

📄

Documentos

Leis, manuais, processos de negócio existentes...

💻

Sistemas Legados

Sistemas atuais que serão substituídos ou integrados.

Passo 2: Elicitação - Como Extrair a Informação?

É um trabalho de detetive. Usamos técnicas diferentes para situações diferentes:

  • Técnicas de Coleta: Entrevistas, Questionários, Workshops.
  • Técnicas de Colaboração: Sessões de brainstorming, design thinking.
  • Técnicas de Observação: Observar o usuário a trabalhar no seu ambiente real.

Passo 3: Resolução de Conflitos

Inevitavelmente, os requisitos entrarão em conflito.

Marketing: "Quero um pop-up gigante na tela inicial!"

vs.

UX Designer: "Quero uma experiência limpa e sem interrupções!"

A ER nos dá técnicas para negociar, priorizar e encontrar um acordo, como votação, análise de alternativas ou compromisso.

Ponto de Verificação: Nossos Casos de Estudo

🚀 Ariane 5

A elicitação falhou em identificar a mudança de contexto como uma fonte de novos requisitos (restrições) para o software reutilizado.

✉️ Sistema Horizon

Falha catastrófica em identificar e elicitar requisitos dos stakeholders corretos (os operadores das agências), resultando num conflito não resolvido entre a visão da gestão e a realidade operacional.

Passo 4: Validação - A Rede de Segurança

É o processo de verificar com os stakeholders se os requisitos documentados são realmente o que eles precisam. É a última chance de encontrar erros antes de escrever uma única linha de código.

  • Técnicas de Revisão: Inspeções formais do documento SRS, Walkthroughs.
  • Técnicas Exploratórias: Usar protótipos para obter feedback, testes Alfa/Beta.

Conclusão: O Ciclo Virtuoso

Documentar e Elaborar não são fases separadas, são um ciclo contínuo.

Elaborar ➔ Documentar ➔ Validar ➔ (Repetir)

Dominar este ciclo é a diferença entre um projeto de sucesso e o próximo "case de desastre" a ser estudado.
